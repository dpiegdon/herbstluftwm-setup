# Mouse-Warp Oscillation Bugs in herbstluftwm

Analyzed and verified against herbstluftwm **v0.9.6-3** (commit `e4712f0a`,
the submodule checkout in this repo). An earlier revision of this document
claimed 0.9.5/`9e50e0e8`; the code is identical in all relevant places, but
line numbers below refer to the v0.9.6-3 tree (with the fix applied, where
noted).

**Status: Bug 1 is the verified root cause. A fix is implemented in the
`herbstluftwm/` submodule (see "The fix" below), together with a regression
test.** Bugs 2 and 3 are real upstream quirks, but they are *not* required
to trigger the oscillation, and this repo's `config/autostart` already
neutralizes both (`theme.minimal.border_width 1`, `frame_gap 0`,
`frame_padding 0`) — which is why the oscillation persisted despite these
config workarounds.

---

## Observed symptom

With `focus_follows_mouse = true` on a multi-monitor setup, the mouse
cursor sometimes starts warping between two monitors repeatedly at high
frequency. It can only be stopped by clicking a mouse button; the click is
then delivered to whichever window the cursor happened to be over, often as
a *drag* rather than a plain click. It is usually correlated with the
cursor crossing windows or screen borders, but sometimes appears to happen
spontaneously.

---

## Bug 1 (root cause): `dropEnterNotifyEvents()` is a no-op during cross-monitor `enternotify()` handling

### Requirements

- `focus_follows_mouse = true`
- Multi-monitor setup
- At least **two** stale `EnterNotify` events for windows on *different*
  monitors in hlwm's event queue (see "How the loop starts" below)

### The defective guard

`XMainLoop::dropEnterNotifyEvents()` (`xmainloop.cpp:239`) flushes all
queued `EnterNotify` events; it is invoked (via the
`MonitorManager::dropEnterNotifyEvents` signal) from
`Monitor::applyLayout()`, `monitor_set_tag()`, `add_monitor`, and — most
importantly — right after the `XWarpPointer()` call in
`monitor_focus_by_index()` (`monitor.cpp:557`), whose whole purpose is to
discard the artificial enter events generated by the pointer warp.

The guard added in commit `62121a9e` ("Don't drop later EnterNotify during
enternotify()", fixing issue #908) makes the function return without
dropping anything while an `EnterNotify` event is being handled:

```cpp
if (duringEnterNotify_) {
    return;   // drops NOTHING
}
```

The commit message justifies this with:

> During the handling of an EnterNotify event, no such artificial enter
> notify events can be created, because only the focus is changed.

**That assumption is wrong for cross-monitor focus changes.** The handler
chain `enternotify()` (`xmainloop.cpp:451`) → `focus_client(c, false,
/*switch_monitor=*/true, false)` (`xmainloop.cpp:481`) →
`monitor_focus_by_index()` (`layout.cpp:642`) runs entirely while
`duringEnterNotify_` is `true`, and it does far more than "change the
focus":

1. `applyLayout()` runs on both the old and the new monitor
   (`monitor.cpp:522-523`): raising, restacking, resizing, and
   showing/hiding decorations — all of which can generate artificial
   `EnterNotify` events if they happen under the cursor. The drop at the
   end of `applyLayout()` is a no-op.
2. If the cursor is not on the new monitor, `XWarpPointer()` is called
   (`monitor.cpp:557`), which *always* generates crossing events. The drop
   right after it is a no-op.

### Why it oscillates (the feedback loop)

A single stale event would only cause a single spurious warp. The sustained
oscillation needs the queue to keep containing an enter event for "the
other" monitor — and the undropped warp events provide exactly that:

```
pointer on A;  queue: [Enter(winB), Enter(winA)]        (two stale events)

process Enter(winB): winB is on B, pointer on A
  → switch to B, warp pointer to B
  → warp generates Enter(B-window); drop is a NO-OP
  queue: [Enter(winA), Enter(B-window)]                  pointer on B

process Enter(winA): winA is on A, pointer on B
  → switch to A, warp pointer to A
  → warp generates Enter(A-window); drop is a NO-OP
  queue: [Enter(B-window), Enter(A-window)]              pointer on A

... repeats forever: every iteration consumes one stale event and the
warp deposits a fresh one, which is processed only after the *next*
queued event has warped the pointer to the opposite monitor again.
```

Two subtle details make this work:

- `enternotify()` ignores events with `ce->focus == true`
  (`xmainloop.cpp:468`), so one might expect the warp-generated enter for
  the newly focused monitor to be filtered out. Even when the X input
  focus is already on the entered client at warp time, the client's
  *decoration* window receives its own (virtual) `EnterNotify` with
  `focus == false` — a decoration is the focus window's ancestor, not an
  inferior — and `Decoration::toClient()` in `enternotify()` happily
  resolves it to the client. So the warp reliably produces a
  focus-change-triggering event.
- The warp destination is the monitor's *saved mouse position* (or its
  center, with `mouse_recenter_gap`), which in practice is virtually
  always over a window or its decoration — that is where users leave the
  pointer. (If the warp happened to land on a bare root-window pixel, the
  resulting enter event would resolve to neither a client nor a frame and
  the chain would stop — this was confirmed experimentally while building
  the regression test.)

### How the loop starts

Bootstrapping needs two stale cross-monitor enter events in the queue,
which **no configuration can prevent**:

- **Crossing the boundary quickly** (A→B→A): the X server queues
  `Enter(winB)`, `Enter(winA2)`; by the time hlwm processes `Enter(winB)`,
  the pointer is back on A → warp → loop. This matches "often related to
  the mouse crossing windows or screen borders".
- **Artificial events from layout changes**: any relayout/restack while
  the pointer is over overlapping windows (max layout, floating clients)
  can generate enter events for windows the user never moved into; the
  drops that should remove them are no-ops whenever the relayout happens
  inside `enternotify()`. This matches "sometimes it just seems to
  happen".

### Why a mouse click stops it (and why it often turns into a drag)

- A `ButtonPress` activates the (synchronous) passive button grab that
  hlwm holds on client windows. While the grab is active, crossing events
  are generated with `mode == NotifyGrab` / `NotifyUngrab`, which
  `enternotify()` ignores (`xmainloop.cpp:453`). That breaks the chain.
- The handler ends with `XAllowEvents(..., ReplayPointer, ...)`
  (`xmainloop.cpp:309`), which delivers the click to whatever window the
  cursor happened to be over mid-oscillation — the observed "the click is
  delegated to one of the windows".
- If the click happens to land on a client *decoration* (border/title),
  `buttonpress()` calls `mouse_initiate_move(client, {})`
  (`xmainloop.cpp:296`), i.e. it starts a window-move drag; on a border
  edge it may start a resize drag instead (`xmainloop.cpp:277`). That is
  the observed "often also sending a drag event". During the oscillation
  the cursor spends a lot of time on freshly-entered decorations, so this
  is common.

### The fix (implemented in the submodule)

The invariant: **after a pointer warp, every `EnterNotify` event still in
the queue describes an outdated pointer position** — both the artificial
events generated by the warp itself and any older stale events. Dropping
them all at warp time is therefore semantically correct even during
`enternotify()`, and it simultaneously breaks the replenishment *and*
drains the bootstrap events. The drop is tied to the warp, not to the
monitor switch.

The `dropEnterNotifyEvents` signal now carries a `force` flag
(`Signal_<bool>`, `monitormanager.h`):

- `monitor_focus_by_index()` emits `true` after `XWarpPointer()`
  (`monitor.cpp`): the drop bypasses the `duringEnterNotify_` guard.
- The layout-change sites (`Monitor::applyLayout()`, `monitor_set_tag()`,
  `add_monitor`) emit `false`: their behavior is unchanged, preserving
  the fix for issue #908 (rapid mouse movement across windows on the same
  monitor must not lose enter events).
- `XMainLoop::dropEnterNotifyEvents(bool force)` skips the drop only
  `if (duringEnterNotify_ && !force)`.

Touched files: `src/monitormanager.h`, `src/monitormanager.cpp`,
`src/monitor.cpp`, `src/xmainloop.h`, `src/xmainloop.cpp`, and a
regression test in `tests/test_basic_events.py`.

**Note on a previously proposed alternative:** an earlier revision of this
document suggested setting a flag when a monitor switch occurs during
`enternotify()` and performing the drop after `enternotify()` returns.
That variant is **wrong**: `focus_client()` calls `applyLayout()` on every
enternotify-triggered focus change (also same-monitor ones,
`layout.cpp:660`), whose signal emission would set the flag and cause a
drop after *every* enternotify — exactly the behavior that commit
`62121a9e` removed to fix issue #908. The drop must be coupled to the
warp, not to the fact that a monitor switch (or relayout) happened.

### Regression test and verification results

`tests/test_basic_events.py::test_no_warp_oscillation_between_monitors`
reproduces the bug deterministically: it sets up two virtual monitors with
one client each, parks the pointer inside each monitor's window so that
the saved mouse positions (= warp destinations) are over the windows, then
SIGSTOPs the hlwm process, performs two non-synced pointer moves (into the
other monitor's window and back) so that exactly the stale event pair from
"How the loop starts" is queued, SIGCONTs hlwm, and asserts that
`monitors.focus.index` converges instead of oscillating.

Verified on this machine (Xvfb, v0.9.6-3):

- **Unpatched** `e4712f0a`: the test **fails** — `monitors.focus.index`
  keeps flipping between `0` and `1` for the whole sampling window, i.e.
  the warp oscillation is live in the test environment.
- **With the fix**: the test passes (5/5 repeated runs, no flakiness),
  and the complete `test_basic_events.py` suite passes (23/23) —
  including `test_enternotify_do_not_drop_events`, the regression test
  for issue #908 whose behavior the fix must preserve.

## Bug 2 (not the active trigger here): `theme.border_width` does not propagate to `theme.minimal`

### Verified code claim

In `theme.cpp:48-50`, the attribute proxies exclude the `minimal` triple:

```cpp
active.makeProxyFor({&tiling.active, &floating.active});
normal.makeProxyFor({&tiling.normal, &floating.normal});
urgent.makeProxyFor({&tiling.urgent, &floating.urgent});
// 'minimal' is NOT a proxy target
```

So `theme.border_width 1` does not set `theme.minimal.border_width`
(default 0). With `smart_window_surroundings = on`, frames with a single
window use the `minimal` scheme (`layout.cpp:280`), giving them a
different effective geometry than multi-window frames — including a
possible 1px mismatch at a monitor boundary.

### Relevance

A geometry mismatch at the boundary makes it *more likely* that small
cursor movements generate competing enter events for different windows,
i.e. it can help bootstrap Bug 1 — but it is neither necessary nor
sufficient for the oscillation. **This config already compensates** with
`hc attr theme.minimal.border_width 1` (`config/autostart`), and the
oscillation still occurred, confirming Bug 1 as the actual cause.

Upstream, making `theme.*` attributes also proxy to `theme.minimal` would
be a behavioral change (the `minimal` theme would no longer be
independently configurable through the parent attributes); a config-level
`theme.minimal.*` assignment is the non-invasive answer.

## Bug 3 (not the active trigger here): asymmetric gaps from `smart_frame_surroundings` / `smart_window_surroundings`

### Verified code claim

`Monitor::applyLayout()` applies `frame_gap` asymmetrically
(`monitor.cpp:171-178`): it insets x/y (left/top) and shrinks
width/height by only one `frame_gap`, relying on `window_gap` to cover the
right/bottom edges — which breaks when `smart_window_surroundings`
suppresses `window_gap` (`layout.cpp:259`). With `smart_frame_surroundings
= hide_all`, a single-frame monitor additionally gets no frame gap at all,
while a split monitor does. Adjacent monitors can therefore expose a 1-2px
strip of frame decoration or root window at the shared boundary.

### Relevance

Same as Bug 2: a possible bootstrap helper for Bug 1, not a cause. **This
config already neutralizes it** with `frame_gap 0` and `frame_padding 0`,
so the preconditions are not met, yet the oscillation occurred anyway.

## Summary

| Bug | Verified? | Role in the oscillation |
|-----|-----------|-------------------------|
| 1   | **Yes — root cause** | Enables the self-sustaining warp loop; fixed in the submodule |
| 2   | Yes (code quirk) | Possible bootstrap helper; already mitigated in `config/autostart`, not required for the bug |
| 3   | Yes (code quirk) | Possible bootstrap helper; already mitigated in `config/autostart`, not required for the bug |

The config-level workarounds for Bugs 2/3 reduce *some* bootstrap
opportunities but cannot prevent the main one (fast boundary crossings),
which is why only the Bug 1 code fix actually resolves the issue.

## Deploying the fix

The fix lives as local changes in the `herbstluftwm/` submodule.
**Careful: `setup.sh` runs `git submodule foreach git clean -dfx` and
`git reset HEAD --hard`, which would wipe these changes.** Either commit
them inside the submodule first, or build manually:

```bash
cd herbstluftwm && mkdir -p build && cd build && cmake .. && make -j$(nproc)
```

Run the relevant tests (needs `xvfb`, `xdotool`, `xterm`, `pytest`,
`python3-xlib`):

```bash
cd herbstluftwm/tests
pytest test_basic_events.py -k 'warp_oscillation or enternotify or drop_enter' -v
```
