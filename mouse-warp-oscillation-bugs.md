# Mouse-Warp Oscillation Bugs in herbstluftwm

Analyzed against herbstluftwm **0.9.5** (commit `9e50e0e8`, current `master`).

---

## Bug 1: `dropEnterNotifyEvents()` is bypassed during cross-monitor `enternotify()` handling

### Requirements

- `focus_follows_mouse = true`
- Multi-monitor setup
- A condition that causes `enternotify()` to trigger a cross-monitor focus change
  (e.g., Bug 2 or Bug 3 below, or an application that places windows overlapping
  monitor boundaries)

### What is triggered

In `xmainloop.cpp:239-252`, `dropEnterNotifyEvents()` has an early-return guard:

```cpp
void XMainLoop::dropEnterNotifyEvents()
{
    if (duringEnterNotify_) {
        return;   // ← drops NOTHING
    }
    XEvent ev;
    XSync(X_.display(), False);
    while (XCheckMaskEvent(X_.display(), EnterWindowMask, &ev)) {
    }
}
```

This guard was introduced in commit `62121a9e` ("Don't drop later EnterNotify
during enternotify()") to fix a problem where rapid mouse movements across
windows on the *same* monitor caused legitimate `EnterNotify` events to be
discarded. The commit message states:

> During the handling of an EnterNotify event, no such artificial enter notify
> events can be created, because only the focus is changed.

**This assumption is incorrect for cross-monitor focus changes.** When
`enternotify()` → `focus_client(c, false, true, false)` triggers
`monitor_focus_by_index()`, the following happens *while `duringEnterNotify_`
is still `true`*:

1. `old->applyLayout()` is called (`monitor.cpp:522`), which rearranges
   windows on the old monitor. At the end it calls
   `dropEnterNotifyEvents.emit()` → **no-op** (early return).

2. `monitor->applyLayout()` is called (`monitor.cpp:523`), which rearranges
   windows on the new monitor. Again `dropEnterNotifyEvents.emit()` → **no-op**.

3. If the mouse is *not* on the new monitor, `XWarpPointer()` is called
   (`monitor.cpp:557`), generating an `EnterNotify` event. Then
   `dropEnterNotifyEvents.emit()` → **no-op**.

After `enternotify()` returns and sets `duringEnterNotify_ = false`, the
**undropped** `EnterNotify` events from steps 1–3 are still in the X event
queue. The main loop picks them up and calls `enternotify()` again — which can
trigger another `monitor_focus_by_index()` in the opposite direction, creating
a **warp oscillation loop**.

### How to reproduce

1. Set `focus_follows_mouse true`, two monitors side-by-side.
2. Create a condition where an `EnterNotify` on monitor B is delivered while
   monitor A is focused (see Bugs 2/3 for concrete triggers).
3. The `enternotify()` handler calls `focus_client()` →
   `monitor_focus_by_index(B)`.
4. If the cursor is warped during this, the resulting `EnterNotify` events
   are *not* dropped.
5. The stale events trigger `enternotify()` again, potentially switching
   back to monitor A.
6. The cycle repeats several times per second until a `ButtonPress` event
   breaks the loop (because button events are handled by a different path
   and don't generate `EnterNotify` chains).

### How to fix

The `duringEnterNotify_` guard needs to be more selective. It should still
prevent dropping legitimate EnterNotify events from real mouse movement, but
it must allow dropping *artificial* events generated by `applyLayout()` and
`XWarpPointer()` during cross-monitor switches.

**Option A (targeted):** Track whether a cross-monitor switch occurred during
`enternotify()`. If yes, perform the drop after `enternotify()` returns:

```cpp
void XMainLoop::enternotify(XCrossingEvent* ce) {
    // ...
    duringEnterNotify_ = true;
    // ... handle the event (may call monitor_focus_by_index) ...
    duringEnterNotify_ = false;
    // NEW: if a monitor switch happened above, now drop artificial events
    if (monitorSwitchOccurredDuringEnterNotify_) {
        monitorSwitchOccurredDuringEnterNotify_ = false;
        dropEnterNotifyEvents();
    }
}
```

**Option B (simpler):** In `monitor_focus_by_index()`, if `XWarpPointer()` is
called, directly inline the event-drop logic (bypassing the
`dropEnterNotifyEvents()` signal and its `duringEnterNotify_` check):

```cpp
XWarpPointer(g_display, None, g_root, 0, 0, 0, 0, new_x, new_y);
// Always drop EnterNotify events from warps, even during enternotify():
XEvent ev;
XSync(g_display, False);
while (XCheckMaskEvent(g_display, EnterWindowMask, &ev)) {}
```

Option A is cleaner because it preserves the event ordering for
same-monitor focus changes.

## Bug 2: Different effective border widths between monitors via `smart_window_surroundings`

### Requirements

- `smart_window_surroundings = on` (or `one_window`)
- Multi-monitor setup
- Different number of windows on each monitor (e.g., 1 window on monitor A,
  2+ on monitor B)
- Any `theme.border_width > 0`

### What is triggered

The `theme.border_width` attribute propagates to `theme.tiling` and
`theme.floating` via proxy (`theme.cpp:48-50`), but **not** to
`theme.minimal`:

```cpp
// theme.cpp:48-50
active.makeProxyFor({&tiling.active, &floating.active});
normal.makeProxyFor({&tiling.normal, &floating.normal});
urgent.makeProxyFor({&tiling.urgent, &floating.urgent});
// Note: 'minimal' is NOT a proxy target
```

The `minimal` `DecTriple` is an entirely separate theme object
(`theme.h:216`). Setting `theme.border_width 1` does *not* set
`theme.minimal.border_width`. The default is **0**.

When `smart_window_surroundings` is active (`layout.cpp:249-256`), frames
with a single visible window get `minimalDecoration = true`, which selects
the `theme.minimal` decoration scheme. So:

| Monitor state          | Decoration scheme    | Effective border width |
|------------------------|----------------------|------------------------|
| 1 window in frame      | `theme.minimal`      | 0 (default)            |
| 2+ windows / split     | `theme.tiling`       | 1 (user-configured)    |

This creates a **1px discrepancy** in the effective window geometry at the
monitor boundary. On the multi-window monitor, the windows are also inset
by `frame_gap` (1px) and `frame_padding` (1px). This misalignment can place
the cursor in a "dead zone" at the monitor boundary where the X server
alternates which decoration window (or root window) is under the cursor
during layout changes — triggering the oscillation from Bug 1.

### How to reproduce

1. Two monitors side-by-side.
2. `smart_window_surroundings on`, `theme.border_width 1`.
3. Monitor A: one tiled window (uses `theme.minimal`, border=0).
4. Monitor B: two tiled windows in a split frame (uses `theme.tiling`,
   border=1, plus frame_gap=1, frame_padding=1).
5. Move the mouse slowly to the shared edge between the monitors.
6. Because of the gap/padding difference, there is a 1–3px zone at the
   boundary of monitor B where the root window or frame decoration is
   exposed (no client decoration covers it).
7. With `focus_follows_mouse`, the cursor alternating between the
   edge of A's full-coverage decoration and B's inset decoration generates
   competing `EnterNotify` events for different windows/frames.
8. Combined with Bug 1, this can start the warp oscillation.

### How to fix

**Option A (config-level):** Explicitly set `theme.minimal.border_width` to
match `theme.tiling.border_width`:

```bash
hc attr theme.minimal.border_width 1
```

**Option B (upstream code fix):** Make `theme.border_width` also propagate to
`theme.minimal`, or make the `DecTriple` proxy chain include the `minimal`
object. In `theme.cpp`, change the proxy targets:

```cpp
active.makeProxyFor({&tiling.active, &floating.active, &minimal.active});
normal.makeProxyFor({&tiling.normal, &floating.normal, &minimal.normal});
urgent.makeProxyFor({&tiling.urgent, &floating.urgent, &minimal.urgent});
```

This would be a **behavioral change** (the `minimal` theme would no longer be
independently configurable via the parent `theme.*` attributes), so it may need
discussion. An alternative is to add a separate propagation mechanism or
document that `theme.minimal` must be configured separately.

## Bug 3: `smart_frame_surroundings` / `smart_window_surroundings` create asymmetric gaps at monitor edges

### Requirements

- `smart_frame_surroundings = hide_all`
- `smart_window_surroundings = on`
- `frame_gap > 0` and/or `frame_padding > 0`
- Multi-monitor with different frame structures per monitor

### What is triggered

The gap/padding suppression logic is conditional on the frame structure:

**Frame gap** (`monitor.cpp:171-178`):

```cpp
if (g_settings->smart_frame_surroundings() == SmartFrameSurroundings::off
    || tag->frame->root_->isSplit()) {
    cur_rect.x += settings->frame_gap();
    cur_rect.y += settings->frame_gap();
    cur_rect.height -= settings->frame_gap();
    cur_rect.width -= settings->frame_gap();
}
```

Frame gap is applied only when `smart_frame_surroundings` is off OR the frame
tree is split. With `hide_all` and a single (non-split) frame, frame gap is
**not applied** → the window fills the monitor area edge-to-edge.

**Window gap + frame padding** (`layout.cpp:258-276`):

```cpp
if (!smart_window_surroundings_active) {
    rect.x += window_gap;
    rect.y += window_gap;
    rect.width -= window_gap;
    rect.height -= window_gap;
    rect.x += frame_padding;
    rect.y += frame_padding;
    rect.width  -= frame_padding * 2;
    rect.height -= frame_padding * 2;
}
```

Window gap and frame padding are applied only when `smart_window_surroundings`
is **not** active.

**Note the asymmetry in the frame_gap application:** It adds `frame_gap` to
x/y (left/top) but only subtracts `frame_gap` from width/height (not
`2*frame_gap`). This means the frame gap is applied to the **left and top**
edges but not to the **right and bottom** edges of the frame tree area. This
is intentional (the `window_gap` handles the right/bottom), but when
`smart_window_surroundings` suppresses `window_gap`, the right/bottom edges
have no gap at all.

The result on adjacent monitors:

- **Single-window monitor (smart active):** Window fills the full content
  area — no gaps on any edge.
- **Multi-window monitor (smart inactive, split frame):** Frame gap on
  left/top (1px), frame padding on all sides (1px each), window gap on
  right/bottom (0px in user config). Total inset from left: 2px (frame_gap
  + frame_padding). Inset from right: 1px (frame_padding only — frame_gap is
  asymmetric and window_gap is 0).

At the shared boundary between monitors, one monitor has windows extending to
the pixel boundary, while the other has a 1–2px gap. This exposed gap
contains the frame decoration window or root window, which can trigger
`EnterNotify` events for a `FrameDecoration` — and the `enternotify()` handler
focuses frames via `root_->focusFrame()` (`xmainloop.cpp:478-485`), which
calls `monitor_focus_by_index()` (`root.cpp:139`).

### How to reproduce

1. Two monitors side-by-side, `frame_gap 1`, `frame_padding 1`.
2. `smart_frame_surroundings hide_all`, `smart_window_surroundings on`.
3. Monitor A: single window (full-coverage, no gaps).
4. Monitor B: split frame with 2+ windows (frame gap + padding applied).
5. Move cursor to the boundary between monitors.
6. On monitor B's side, the 1–2px gap exposes the frame window.
7. With `focus_follows_mouse`, entering this frame window triggers
   `root_->focusFrame()` → `monitor_focus_by_index()`.
8. This generates a cross-monitor switch within `enternotify()`, hitting Bug 1.

### How to fix

**Option A (config-level):** Set `frame_gap 0` and `frame_padding 0`, or
set `smart_frame_surroundings off` and `smart_window_surroundings off` to
get consistent borders everywhere.

**Option B (upstream):** When `smart_frame_surroundings` suppresses the frame
gap, also ensure the contained window fills the area symmetrically. Or make
frame_gap symmetric (apply `2*frame_gap` to width/height instead of
`1*frame_gap`). The current asymmetric application relies on window_gap
covering the other edges, which breaks when window_gap is suppressed.

## Summary and interactions

| Bug | Severity | Standalone? | Interaction |
|-----|----------|-------------|-------------|
| 1   | **High** — core event-loop defect | Needs a trigger (Bug 2 or 3) | Enables the oscillation loop |
| 2   | **Medium** — wrong default | Can cause visual inconsistency alone | With Bug 1 → oscillation |
| 3   | **Medium** — gap asymmetry | Cosmetic alone | With Bug 1 → oscillation |

**The oscillation occurs when Bug 2 or Bug 3 provides the trigger (competing
`EnterNotify` events at the monitor boundary) and Bug 1 prevents the
event-dropping safety mechanism from working.**

The button-press breaking the cycle is explained by the `buttonpress()`
handler (`xmainloop.cpp:258`) calling `XAllowEvents(ReplayPointer)` and the
mouse manager consuming the event — neither path generates `EnterNotify`
chains. The occasional drag behavior is caused by `buttonpress()` calling
`mouse_initiate_move()` or `mouse_initiate_resize()` for the window the
cursor happened to be over at the moment the user clicked to break the loop.

### User config workaround (immediate)

Add to `config/autostart`:

```bash
hc attr theme.minimal.border_width 1
hc set frame_gap 0
hc set frame_padding 0
```

This ensures identical effective borders and gaps on all monitors, eliminating
the trigger conditions for the oscillation.
